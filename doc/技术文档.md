# 农产品溯源系统 - 技术文档

## 一、系统概述

本系统是一套基于区块链技术的农产品全链路溯源平台，覆盖从种植/养殖、检疫质检、加工宰杀、仓储、物流运输到销售的完整供应链流程。系统由三端组成：PC 管理后台（Vue.js）、微信小程序（原生框架）、Spring Boot 后端服务，并集成以太坊区块链（Ganache 本地测试网）实现数据不可篡改存证。

---

## 二、技术栈

### 后端
- Spring Boot 2.7.18 + Java 11
- MyBatis Plus 3.5.3.1（ORM 框架，逻辑删除、分页插件）
- MySQL 8.0 + Druid 连接池
- Web3j 4.9.8（以太坊 Java SDK）
- JWT（jjwt 0.11.5，无状态鉴权）
- Hutool 5.8.23（工具库）
- Spring AOP（操作日志切面）
- Spring Security Crypto（密码加密）

### 小程序
- 微信原生小程序框架（WXML + WXSS + JS）
- 统一请求封装（utils/request.js）
- 环境配置（config.js，支持 dev/prod 切换）

### 区块链
- Ganache 2.7+（本地以太坊测试网，RPC: http://127.0.0.1:8545）
- Solidity 0.8.x 智能合约（Traceability.sol）
- Remix IDE（合约编译与部署）

## 五、后端模块说明

后端服务监听端口 **8888**，所有业务接口统一前缀 `/api`，认证接口前缀 `/auth`。统一响应格式为 `Result<T>{ code, message, data }`。

### 5.1 认证模块（AuthController）

路径前缀：`/auth`

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | /auth/login | 用户名密码登录，返回 JWT Token 和用户信息，同时写入登录日志 |
| POST | /auth/register | 普通用户注册（消费者账号，user_type=3） |
| GET  | /auth/info | 根据 Token 获取当前用户信息（Token 有效性验证） |
| POST | /auth/logout | 退出登录（前端清除 Token，后端无状态，无需处理） |

**AuthService 核心逻辑：**

`login(LoginDTO)` 执行步骤：
1. 按 username 查询 sys_user，若不存在或 status=0 则返回错误
2. 使用 BCrypt 比对密码，不匹配则返回错误
3. 生成 JWT Token（有效期 7 天，携带 userId 和 username），通过 `Authorization: Bearer <token>` 请求头传递
4. 查询关联的 enterprise_info（若 enterprise_id 不为空），将 enterprise_type、audit_status 一并返回
5. 写入 sys_login_log（登录时间、IP、设备信息）

`registerOrdinaryUser(body)`：创建 user_type=3 的普通用户，密码 BCrypt 加密，不绑定企业。

路径前缀：`/api/batch`

| 方法 | 路径 | 功能 |
|------|------|------|
| GET  | /api/batch/list | 分页查询批次列表（支持多状态、关键词、企业类型、是否上链等多维筛选） |
| GET  | /api/batch/status-count | 查询各状态批次数量统计 |
| GET  | /api/batch/detail | 查询批次详情（支持 batchCode 或 id 两种方式，batchCode 优先） |
| POST | /api/batch/create | 创建新批次（同时支持 /add 路径），创建后自动上链存证 |
| DELETE | /api/batch/delete/{id} | 删除批次（仅允许状态为 1-初始化 或 4-加工中的批次） |
| GET  | /api/batch/quarantine-check/{batchCode} | 查询批次检疫合格状态（加工企业扫码前调用） |
| POST | /api/batch/receive | 扫码接收批次（前置条件：必须有 check_result=1 的检疫合格记录） |
| POST | /api/batch/cancel-receive/{batchId} | 取消接收批次（状态回滚为 3） |
| POST | /api/batch/reject/{batchId} | 拒绝接收批次（状态回滚为 6，删除对应运输记录，通知养殖企业） |

**批次状态流转（完整）：**

```
1(初始化) -> 2(生长中) -> 3(已收获/待检疫)
                              |
                         [提交检疫] -> check_result=1 -> 5(已检疫)
                                                          |
                                                    [加工企业接收] -> 4(加工中)
                                                                       |
                                                                  [创建加工记录] -> 9(加工完成)
                                                                                     |
                                                                               [入库] -> 6(已入库)
                                                                                          |
                                                                                    [运输] -> 7(运输中)
                                                                                               |
                                                                                         [销售] -> 8(已销售)
```

状态说明：
- 1：初始化（批次刚创建）
- 2：生长中（有过程记录后）
- 3：已收获/待检疫
- 4：加工中（加工企业已接收）
- 5：已检疫（检疫合格）
- 6：已入库
- 7：运输中
- 8：已销售
- 9：加工完成（ProcessingService.createProcessing 执行后自动设置）

**BatchService 核心逻辑：**

`createBatch(params)`：
1. 优先使用前端传入的 batchCode，否则自动生成（`BATCH` + yyyyMMddHHmmss + 4位随机数）
2. 插入 trace_batch，初始 batch_status=1
3. 生成二维码 URL：`http://api.domain.com/trace?code={batchCode}`
4. 计算 SHA-256 哈希（`BATCH_INIT|code=...|product=...|qty=...|enterprise=...|ts=...`），traceId 格式为 `{batchCode}_INIT`
5. 调用 Web3jUtils.uploadHash() 上链，保存 tx_hash、block_number、chain_time
6. 发送消息通知给所有管理员

`receiveBatch(dto)`：
1. 校验批次状态（必须为 3/5/6/7 之一）
2. 校验 trace_inspection 中存在 check_result=1 的记录，否则拒绝
3. 更新 batch_status=4，写入 receiver、receive_date、receive_enterprise_id

`rejectBatch(batchId, reason, rejectEnterpriseId)`：
1. 校验批次状态必须为 7（运输中）
2. 回滚 batch_status=6（已入库）
3. 删除指向该加工企业的 trace_transport 记录
4. 向养殖企业发送业务消息通知

**状态统计（getStatusCount）：**
- 养殖企业视角：breeding（状态1+2）、pending（状态3）、completed（状态4-9）
- 加工企业视角：pendingReceive（状态5+7，目标为本企业）、processing（状态4+9）、completed（状态6+8+9）

| 方法 | 路径 | 功能 |

| DELETE | /api/quarantine/apply/delete/{id} | 删除检疫申请（待检疫记录物理删除；已完成记录回滚 check_result 并将批次状态回退为 3） |
| GET  | /api/quarantine/apply/list | 查询检疫申请列表（养殖企业视角） |

**QuarantineService 核心逻辑：**

`applyQuarantine(params)`：
1. 查找批次（batchCode 或 batchId）
2. 创建 trace_inspection 记录，check_result=null（待检疫状态）
3. 若未指定 inspection_enterprise_id，自动查找 enterprise_type=3 且已审核通过的第一家检疫机构
4. 更新 batch_status=3（已收获/待检疫）
5. 本步骤不上链，上链在 submitQuarantine 时执行

`submitQuarantine(dto)`：
1. 查找该批次 check_result=null 的待检疫记录，有则更新，无则新增
2. 填写 check_result（1-合格/0-不合格）、inspector、cert_no、cert_image
3. 检疫合格时：更新 batch_status=5（已检疫）
4. 调用 performChainUpload() 上链

`performChainUpload(inspection, batch)`：
1. 构建哈希字符串（包含批次信息 + 检疫结果 + 证书信息），计算 SHA-256
2. traceId 格式：`{batchCode}_INSPECTION_{inspectionId}`
3. 调用 Web3jUtils.uploadHash() 上链，保存 tx_hash、block_number、chain_time
4. 调用 BlockchainTransactionService.recordTransaction() 记录交易
5. 发送上链成功消息通知给检疫机构

### 5.10 溯源查询模块（TraceController）

路径前缀：`/api`

| 方法 | 路径 | 功能 |
|------|------|------|
| GET  | /api/trace/detail | 获取溯源详情（聚合批次信息 + 全生命周期时间轴 + 企业信息），同时写入 user_scan_log |

**核心方法**：

`onLaunch()`：启动时依次调用 checkLogin() 和 checkUpdate()

`checkLogin()`：从本地缓存（wx.getStorageSync）恢复 token、userInfo、userRole、enterpriseType，恢复后异步调用 validateToken() 验证 Token 有效性

`validateToken()`：请求 `GET /auth/info`，成功则刷新 globalData 和缓存；Token 失效（code != 200）则静默调用 clearLoginInfo()；网络异常时保留本地缓存（不强制登出）

`setLoginInfo(data)`：登录成功后将 token、userInfo、userRole、enterpriseType 同时写入 globalData 和 wx.setStorageSync

`clearLoginInfo()`：清除 globalData 和本地缓存中的所有登录状态

`requireLogin()`：检查 token + userInfo 是否存在，未登录则弹窗引导跳转 `/pages/login/login`，返回 false

`resolveUserRole(userInfo)`：角色推导规则：
- userType=1 → "admin"
- userType=2 且 auditStatus=1 → "enterprise"
- userType=2 且 auditStatus!=1 → "enterprise-review"
- userType=3 → "consumer"

`refreshUserInfo(callback)`：重新请求 `/auth/info` 刷新用户信息，完成后执行回调（各页面 onShow 时调用）

`checkUpdate()`：调用 wx.getUpdateManager 检测小程序更新，有新版本时弹窗提示重启

## 七、区块链集成说明

### 7.1 智能合约（Traceability.sol）

合约部署在 Ganache 本地以太坊测试网（chainId: 1337，RPC: http://127.0.0.1:8545）。

核心函数：
- `uploadHash(string traceId, string dataHash) returns (bool)`：将溯源ID和数据哈希写入链上存储，记录操作者地址和时间戳
- `getHash(string traceId) returns (string, string, address, uint256)`：查询链上记录，返回 traceId、dataHash、operator、timestamp

### 7.2 数据上链流程

每个业务环节（批次创建、过程记录、检疫、加工、仓储、运输、销售）在数据写入 MySQL 后，执行以下上链流程：

1. 将业务数据序列化为 JSON 字符串
2. 计算 SHA-256 哈希值（dataHash）
3. 以 `业务类型_业务ID` 为 traceId，调用 `Web3jUtils.uploadHash(traceId, dataHash)`
4. 获取交易哈希（txHash），等待交易回执获取区块号（blockNumber）
5. 将 txHash、blockNumber、chainTime、dataHash 回写到对应业务表
6. 调用 `BlockchainTransactionService.recordTransaction()` 保存交易记录，更新日 Gas 统计

### 7.3 溯源验真流程

消费者扫码后，在溯源详情页点击"区块链验真"：

1. 前端调用 `POST /api/blockchain/verify`，传入 batchId
2. 后端查询该批次所有有 txHash 的环节记录
3. 对每个环节，重新计算当前数据库数据的 SHA-256 哈希
4. 调用 `Web3jUtils.verifyHash(traceId, currentDataHash)` 与链上哈希比对
5. 返回每个节点的验证结果（verified/txHash/nodeName）
6. 全部节点验证通过则 verified=true，表示数据未被篡改

### 7.4 区块链配置项（application.yml）

```yaml
blockchain:
  rpc-url: http://127.0.0.1:8545        # Ganache RPC 地址
  contract-address: 0x...               # 已部署的合约地址（必须配置）
  account-address: 0x...                # Ganache 解锁账户地址（开发环境）
  gas-limit: 3000000                    # Gas 上限
  gas-price: 20                         # Gas 价格（Gwei）
```

| 现象 | 原因 | 解决方案 |
|------|------|---------|

### 9.2 角色关系说明

本系统共有 5 类用户身份，每类身份对应不同的操作权限和小程序界面。

**一、身份是怎么来的？**

用户身份由注册/审核流程决定，分三条路径：

```
路径 A：直接注册普通用户
  填写用户名密码注册 -> 身份：消费者
  可以：扫码查看溯源信息，无法进行任何业务操作

路径 B：企业入驻申请
  提交企业资料 -> 等待管理员审核 -> 审核通过
  审核通过后身份自动升级为对应企业类型，解锁业务功能

路径 C：平台管理员
  数据库初始化时预置，不通过注册流程创建
```

**二、五类身份及其权限**

| 身份 | 如何获得 | 小程序工作台显示内容 |
|------|---------|-------------------|
| 消费者 | 普通注册 | 无工作台，仅可扫码溯源 |
| 审核中企业 | 已提交入驻申请，尚未通过 | 显示审核进度，被驳回可重新提交 |
| 种植养殖企业 | 企业入驻审核通过（类型=养殖/种植） | 批次管理、过程记录、检疫申请、区块链存证 |
| 加工宰杀企业 | 企业入驻审核通过（类型=加工宰杀） | 扫码接收批次、加工记录、仓储、运输、销售 |
| 检疫质检企业 | 企业入驻审核通过（类型=检疫质检） | 检疫申报列表、检疫结果录入、报告上传 |
| 平台管理员 | 数据库预置 | 企业审核、用户管理、系统配置、区块链管理 |

**三、企业审核通过时系统自动完成的操作**

管理员点击"审核通过"后，后端自动执行以下三步，无需手动操作：

1. 将企业状态标记为"已通过"
2. 将用户账号与该企业绑定，同步企业类型（种植养殖/加工宰杀/检疫质检）
3. 为用户分配对应角色，解锁相应业务功能
